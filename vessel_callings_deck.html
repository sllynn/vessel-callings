Who was where, and when?

A vessel calling is a record of a ship entering and leaving a geographic zone — a port, an EEZ, a Traffic Separation Scheme lane, an ocean basin.

Callings are foundational to maritime intelligence. They power:

  • Sanctions monitoring — did this vessel call at a restricted port?
  • Carbon & emissions reporting — which EEZ saw what traffic?
  • Port state control — compliance with routing measures
  • Fleet analytics — dwell times, transit patterns, schedule adherence

Marine Intelligence Provider A

Global maritime intelligence firm. Tracks the full global fleet against a rich catalogue of geographic zones — from ocean basins to individual berths.

The ask

Continuously derive callings from a live AIS feed. Keep them correct as positions arrive late and zone boundaries change.

Naive approach

For every incoming position, run ST_Contains against every shape.
That's 10M × 150K = 1.5 trillion geometry tests per hour. A geometry test takes microseconds — this still takes thousands of CPU-hours.

What we need

A way to index both shapes and positions so the join is an integer equality test — fast — and geometry is invoked only where it's truly needed.

Deep Dive 1 of 4

H3: a hexagonal grid over the globe

H3 (Uber, open source) divides the Earth into hierarchical hexagonal cells. Each cell has a unique 64-bit integer ID — cheap to compute and compare.

Hexagons are used because they have a uniform distance from centre to edge. Unlike squares, all six neighbours are equidistant — important for spatial queries.

H3 supports 15 resolutions. The pattern covers the full zone hierarchy:

Zone typeH3 resApprox. cell area
Ocean basins286,745 km²
EEZs (global)5252 km²
IMO TSS / routing7–80.7–5 km²
Port areas11~2,150 m²
Berths13~44 m²

h3_tessellateaswkb

Databricks native SQL function. Decomposes a geometry into the set of H3 cells that cover it at a given resolution. Each cell is stored as a row in shape_cells:

SELECT
  shape_id,
  cell,       -- BIGINT H3 cell ID
  target_res,
  core,       -- fully interior?
  chip_wkb    -- boundary polygon chip
FROM h3_tessellateaswkb(geom, target_res)
Deep Dive 1 of 4

Core vs. boundary cells — the chip pattern

Core cells — fast path

The hexagon is entirely inside the polygon. Any vessel position in this cell is guaranteed to be inside the shape. The equi-join on cell ID is the only test needed. No geometry required.

Boundary cells — chip refinement

The hexagon straddles the polygon edge. A vessel could be inside or outside. We store the chip — the intersection of the hexagon with the polygon — and run ST_Contains(chip, point) only on these cells.

Result in practice

Boundary cells are a thin ring around the polygon perimeter. For large shapes (EEZs, ocean basins), the vast majority of matches hit core cells — geometry runs on a small fraction of rows.

in the Multi-Resolution Join

Deep Dive 2 of 4

h3_toparent() vs h3_longlatash3() — why it matters

We need to join positions against shapes at multiple resolutions simultaneously. The naive approach walks the H3 index hierarchy:

-- WRONG: index-walked ancestor
LATERAL VIEW inline(array(
  named_struct('cell', h3_toparent(h3_cell, 2), 'res', 2),
  named_struct('cell', h3_toparent(h3_cell, 5), 'res', 5),
  named_struct('cell', h3_toparent(h3_cell, 8), 'res', 8)
)) a
-- RIGHT: geometric lookup at each resolution
LATERAL VIEW inline(array(
  named_struct('cell', h3_longlatash3(lon, lat, 2), 'res', 2),
  named_struct('cell', h3_longlatash3(lon, lat, 5), 'res', 5),
  named_struct('cell', h3_longlatash3(lon, lat, 8), 'res', 8)
)) a

Why h3_toparent() fails

H3 uses aperture-7 subdivision: each child hexagon is rotated ~19.1° relative to its parent. Child cells "stick out" beyond their parent's boundary into neighbouring parents' areas.

Consequence: for a position near a res-N cell boundary,
h3_toparent(h3_cell, N) ≠ h3_longlatash3(lon, lat, N)
They can be different cells.

Gaps-and-islands: deriving callings from a position stream

We need to find contiguous runs of positions where vessel V is inside shape S — each run is a calling. The classical SQL trick:

pos_num − match_num is constant inside a contiguous run, and increments when an "out" position interrupts the run.

event_tsmatchpos_nummatch_numdiff
T1—1——
T2A2A:11
T3—3——
T4A4A:22
T5B5B:14

Three distinct diff values → three callings: two for A (both closed), one for B (open).

-- 1. Global per-vessel position number
row_number() OVER (
  PARTITION BY vessel_id
  ORDER BY event_ts
) AS pos_num,

-- 2. Per-(vessel, shape) match number
row_number() OVER (
  PARTITION BY vessel_id, shape_id
  ORDER BY event_ts
) AS match_num,

-- 3. The magic: group by the diff
GROUP BY
  vessel_id, shape_id,
    (pos_num - match_num)

Why not a gap threshold?

The original design used a time-gap heuristic. Rejected by the customer: vessels can sit in a shipyard for months. Any reasonable gap would close callings prematurely. The gaps-and-islands approach is purely observational — a calling closes when we see the vessel outside.

Deep Dive 3 of 4

Handling late-arriving positions correctly

AIS is patchy. Messages arrive late or out of order. A late OUT-position can close — or even split — an existing calling that was derived from earlier batches.

Each batch re-derives callings only for the affected (vessel, shape) pairs — the candidate_pairs CTE:

candidate_pairs AS (
  -- (a) new IN-matches in this batch
  SELECT DISTINCT vessel_id, shape_id
  FROM batch_matches

  UNION

  -- (b) open callings for vessels with ANY
  --     new position (late OUT may close them)
  SELECT c.vessel_id, c.shape_id
  FROM callings_gold c
  WHERE c.exit_ts IS NULL
    AND c.vessel_id IN (
      SELECT vessel_id FROM batch_positions)

  UNION

  -- (c) closed callings whose window contains
  --     a new batch position (split case)
  SELECT c.vessel_id, c.shape_id
  FROM callings_gold c
  JOIN batch_positions p ON p.vessel_id = c.vessel_id
   AND p.event_ts BETWEEN c.entry_ts AND c.exit_ts
)

exit_ts semantics

When closed, exit_ts = the event_ts of the first observed out-position after last_seen_ts. This is an upper bound — the true exit happened somewhere in (last_seen_ts, exit_ts]. Both endpoints are preserved on the row for downstream analytics.

Retraction window

Lookback is capped at a configurable retraction_window (default 7 days). Late positions beyond the window are still stored in Bronze/Silver and queryable ad-hoc — but the streaming MERGE won't retract callings for them. A separate replay job handles older corrections.

Atomic writes

BEGIN ATOMIC … END (DBR 17+): MERGE for updates/inserts + DELETE for orphans in a single transaction. The front-end never sees an inconsistent intermediate state.

The Streaming MERGE
& Atomic Updates

Deep Dive 4 of 4

The atomic update — MERGE + DELETE in one transaction

BEGIN ATOMIC

  -- Update existing callings + insert new ones
  MERGE INTO callings_gold AS c
  USING callings_from_diff AS n
  ON  c.vessel_id = n.vessel_id
  AND c.shape_id  = n.shape_id
  AND c.entry_ts  = n.entry_ts
  WHEN MATCHED THEN UPDATE SET
      last_seen_ts = n.last_seen_ts,
      exit_ts      = n.exit_ts,
      n_positions  = n.n_positions,
      as_of_ts     = current_timestamp()
  WHEN NOT MATCHED THEN INSERT (
      vessel_id, shape_id, entry_ts,
      last_seen_ts, exit_ts, n_positions, as_of_ts
  ) VALUES (n.vessel_id, ...);

  -- Remove orphans the re-derivation no longer
  -- produces — scoped to candidate_pairs only
  DELETE FROM callings_gold
   WHERE EXISTS (candidate_pairs scope)
     AND NOT EXISTS (callings_from_diff);

END

Why the DELETE scope matters

Without scoping the DELETE to candidate_pairs, the query reads "not in callings_from_diff" as "orphan" — and silently deletes valid historical callings for shapes the vessel passed through in earlier batches.

The fix: only delete rows for (vessel, shape) pairs the current batch is actively re-deriving. Shapes outside the batch scope are untouched.

What the pattern delivers

  • Correctness under late arrival — positions delayed by hours still close or split callings correctly via candidate_pairs re-derivation
  • Shape mutation replay — a redrawn boundary triggers CDF-driven recompute; prior callings remain queryable via Delta time travel
  • Product-native spatial functions — H3 functions (h3_*, h3_tessellateaswkb) GA from DBR 16+; geometry functions (ST_Contains, ST_GeomFromWKB) GA from DBR 18.3
  • Full zone hierarchy in one join — ocean basins through berths (res 2–13) handled by a single equi-join pass; no per-resolution pipeline variants
  • Databricks Asset Bundle — deploy to dev or prod target with a single databricks bundle deploy
  • Scales to the full fleet — re-derivation is per-(vessel, shape) pair and naturally parallelised across workers; capped lookback bounds batch cost
